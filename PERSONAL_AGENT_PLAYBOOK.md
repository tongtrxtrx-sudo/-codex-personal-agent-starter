# 个人智能体使用手册

## 目的

这个仓库已经不是一个通用 demo，而是一个面向 AI 开发与设计工作的个人智能体工作台。

这份手册用于指导你：

- 正确启动 Codex 会话
- 用好当前仓库里的 skills
- 从真实使用中持续培育这个智能体
- 判断什么时候该修改规则、记忆或 skill

## 系统结构

可以把这套系统理解为四层：

1. ChatGPT 是长期驾驶舱，负责更宽的上下文、笔记和持续思考。
2. Codex 是本地执行层，负责文件、代码、脚本和命令行任务。
3. `AGENTS.md` 负责定义这个仓库里的持久行为规则。
4. `memory/` 和 `.agents/skills/` 负责在不平台化的前提下完成个性化。

这套设计是刻意保持轻量的。优先使用本地 skill、短提示和小脚本，而不是大型编排系统。

## 核心文件

- `AGENTS.md`
  仓库级行为规则、安全边界和执行方式。

- `memory/profile.md`
  稳定的个人偏好、工作风格、质量标准和输出默认值。

- `memory/current_focus.md`
  当前周或当前阶段的短期重点。

- `inbox/capture.md`
  原始想法、待办、决定和碎片内容的收件箱。

- `.agents/skills/quick-capture`
  用于快速记录，不做过度展开。

- `.agents/skills/morning-brief`
  基于本地文件生成简短开工简报。

- `.agents/skills/file-triage`
  以 dry-run 为先的安全文件整理。

## 如何启动

在仓库根目录执行：

```powershell
cd D:\work\myclaw\codex-personal-agent-starter
npx @openai/codex@latest
```

项目级 `.codex/config.toml` 已经默认使用 `gpt-5.4`。

进入 Codex 交互界面后，要把自然语言输入到 Codex 的输入区，而不是 PowerShell 提示符里。

## 每天开工第一句

推荐默认开场白：

```text
仅基于当前本地项目文件，用一段话告诉我今天最值得推进的事、第一步该做什么，以及是否有任何高风险动作需要我确认。
```

这样可以让会话从当前本地状态出发，而不是一上来就做空泛规划。

## 推荐的日常工作流

### 1. 先记录

当想法还粗糙时，不要过度说明。

```text
用 quick-capture 记下：
<原始想法、任务、决定、链接或提醒>
```

适合的场景：

- 想法
- 待办
- 决定
- 链接
- 会议记录

### 2. 再简报

在开工、切换上下文或卡住时，用晨报类提示。

```text
用 morning-brief 基于当前本地文件给我一个超短工作简报。
如果上下文不足，要明确说出来。
```

### 3. 再执行

任务一旦明确，就给 Codex 一个具体指令，让它在这个仓库里工作。

示例：

```text
检查相关文件，用一段话告诉我最合适的下一步实现动作。
```

```text
把这个模糊想法压成今天能做的 3 个具体动作，保持简短。
```

```text
阅读相关文件，做最小且安全的修改来解决这个问题。
```

### 4. 谨慎整理文件

`file-triage` 一律先 dry-run。

```text
显式使用 $file-triage，对 D:\work\myclaw 做 dry-run，只展示计划。
```

确认无误后再 apply：

```text
显式使用 $file-triage，只执行上一次已确认 dry-run 里的那些移动。
```

## 适合本仓库的提示词形态

这些提示词形态最适合：

- “用 quick-capture 记下：...”
- “用 morning-brief 基于当前本地文件 ...”
- “只检查当前任务必要的文件，然后告诉我最好的下一步。”
- “做最小且安全的改动。”
- “先 dry-run，再展示计划。”
- “更新你的规则：...”

这些提示词形态不适合：

- 一上来就要求做大型平台重构
- 在本地工作流尚未稳定前要求自动化一切
- 把战略、实现、自动化和多个无关想法混在一个超长提示里

## 如何培育这个智能体

这个智能体应该从重复的真实工作中长出来，而不是从抽象架构想象中长出来。

### 什么时候改 `memory/profile.md`

当这些东西发生稳定变化时改：

- 长期工作方式
- 默认回答风格
- 质量标准
- 主要任务类型

例如：

- 完整性比速度更重要
- 你想要更短或更长的默认回答
- 工作从实现偏重转向设计偏重

### 什么时候改 `memory/current_focus.md`

当这些东西发生变化时改：

- 这周重点变了
- 当前项目变了
- 你希望晨报反映新的优先级顺序

这个文件应该始终保持短而新鲜。

### 什么时候改 `AGENTS.md`

当这些问题反复出现时改：

- 智能体重复犯同一种行为错误
- 某条安全规则需要固化
- 仓库工作流发生变化
- 你想强化默认操作方式

`AGENTS.md` 应该保持实用，写稳定行为，不写暂时细节。

### 什么时候新建或修改 skill

出现以下情况时考虑做 skill：

- 同一类任务重复出现 2 到 3 次
- 工作流需要稳定结构
- 一个小脚本能明显提升可靠性
- 这个任务有清晰触发语和清晰输出形态

好的 skill 通常具备：

- 只做一件事
- 触发条件明确
- 输入范围清晰
- 输出格式稳定

## 培育闭环

建议你按这个闭环持续迭代：

1. 在仓库里运行真实任务。
2. 观察摩擦、跑偏和重复纠正。
3. 判断修正应该落在哪：
   - `profile.md`：稳定个人偏好
   - `current_focus.md`：短期重点
   - `AGENTS.md`：持久仓库行为
   - `.agents/skills/`：可重复工作流
4. 做一个小修改。
5. 立刻再跑一个真实任务验证。

不要一次改很多行为规则。小而经过验证的改进，比大而泛的重写更有效。

## 每周维护建议

每周做一次简短维护：

1. 清掉 `memory/current_focus.md` 里过期的重点。
2. 把已经稳定的做法提升到 `memory/profile.md` 或 `AGENTS.md`。
3. 看看是否有重复提示词值得升级成 skill。
4. 清理一切增加负担但没有明显价值的东西。

## 培育时的安全原则

- 没有真实重复需求前，不要加 API、后台服务或 webhook 自动化。
- 文件整理不允许跳过 dry-run。
- 在本地工作流稳定之前，不要把系统做成平台。
- 暂时性的实验不要过早写成长期规则。

## 图片输入建议

如果 Windows 下 Codex 终端里的图片粘贴不稳定，优先改用文件输入，而不是直接贴剪贴板图片。

更稳的做法：

1. 先把截图保存成 PNG，再拖入 Codex 输入区。
2. 或者直接带图片文件启动：

```powershell
npx @openai/codex@latest -i D:\work\myclaw\screen.png
```

这通常比直接粘贴剪贴板图片更稳定。

## 下一步建议

后续的改进应来自真实使用，而不是抽象猜测。

高价值的下一步：

- 连续一周每天用 `quick-capture`
- 每次真实开工前都用一次 `morning-brief`
- 在多次 dry-run 后再收紧 `file-triage`
- 只有当某类工作真正重复出现时，才新增 skill

## 一句话原则

让这个智能体从真实工作中长出来，规则保持短，优先让它更有用，再让它更复杂。
